關於我

我用一根手指打字。所以我特別在意效率、自動化、以及技術的本質。

一個有趣的邊界案例

在討論「vibe coding」時,Theo 把開發者分成兩類:

  • Type A: 有基礎,用 AI 加速
  • Type B: 完全依賴,不懂原理

我是個有趣的 edge case:我有 Type A 的理解能力(架構設計、技術決策、系統除錯),但因物理限制,必須用接近 Type B 的工作流程(高度依賴 AI 執行)。

這讓我對「AI 協作」有獨特視角:創作的核心不在於親手執行,而在於意圖管理、品質把關、技術決策。

核心信念

技術中立

VID/PID 是公開資訊、付費牆問題主要在網站設計、KVM 偽裝不等於作弊——這些是技術事實,不該因商業或政治壓力被迴避。工具就是工具。

效率是尊重

減少點擊、減少輸入、自動化一切可自動化的事——不是偷懶,是對時間成本的尊重。特別是當每次打字都有代價時。

拒絕保護性歧視

我不需要「為你好」的限制。我需要工具、選擇、平等對待——不是同情,不是保護,不是「我們幫你決定什麼對你好」。

務實主義

重要的是決策對不對,不是決策怎麼產生的。重要的是觀點有沒有道理,不是觀點怎麼被表達的。

為什麼值得讀

因為這裡的視角不太一樣。

物理限制沒讓我變得保守,反而讓我更堅持「技術應該自由」。因為我知道工具被限制 = 生活被限制。因為我拒絕「保護性限制」。

這種經歷讓我對技術本質有更清晰認識:技術的核心不是「能親手做」,而是「理解系統、做出決策、解決問題」。

為什麼寫這個部落格

在這裡,我想表達最真實的觀點與聲音。

現實中需要顧慮的太多——別人的感受、社會的眼光、場合的適當性。平常偽裝的夠多了。

這裡是我可以誠實表達的空間:

  • 技術就是技術,不用包裝成勵志故事
  • 觀點就是觀點,不用為了和諧而妥協
  • 問題就是問題,不用假裝一切都好

我不想寫「克服困難」的勵志文,我想討論技術的本質。
我不想展現「正能量」,我想展現真實的思考。
我不想被定義成「身障者做到某某事好厲害」,我想被看見的是想法本身。

所以這裡會有尖銳的觀點、會有技術的批判、會有社會不太談的議題。
也會有真實的掙扎、困惑、以及不完美。

因為這才是真實的。

關於這個網站

b-log 是一個雙重用途的內容系統，完全存在於 GitHub 儲存庫中。它既是分享見解的部落格，也是管理項目的待辦清單。每篇文章都是一個 Markdown 檔案，一個在 data/posts.json 中的小型條目，僅此而已。

為什麼適合 AI 協作

  • Prompts 很簡單：「在 content/posts 建立 Markdown 檔案並更新目錄」
  • 無需建置步驟或工具鏈 — GitHub Pages 直接發布靜態檔案
  • 結構保持一致，AI 代理可以放心擴展或重構
  • 支援 WordPress 風格的永久連結（/category/slug/），自動化生成重定向頁面

網站技術棧

純 HTML/CSS/JS + Markdown，部署在 GitHub Pages，使用 GitHub Actions 自動生成 WordPress 風格永久連結。

如何發布文章

  1. 在 data/posts.json 中複製一個條目，修改 slug
  2. 在 content/posts/ 中建立同名的 Markdown 檔案
  3. 提交、推送，GitHub Pages 會自動處理其餘工作
  4. GitHub Actions 會自動生成 /{category}/{slug}/ 格式的永久連結
